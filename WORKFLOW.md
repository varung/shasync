# shasync — working with your files across machines

This document describes the intended day-to-day flow: who runs what commands
when, what happens on a clean sync, what happens when two machines edit the
same file at the same time, and what the guard-rails are. It's written for
the single-user-many-machines case (laptop + desktop + workstation) that
shasync is tuned for. For the internals — how manifests are stored, how
encryption works, how the remote is laid out — read `ARCHITECTURE.md`.

---

## 1. The mental model

shasync has three layers of state. Two are on your machine:

- **Working tree** — the actual files in your project directory.
- **Local HEAD** — a pointer at `.blobs/HEAD` to the manifest that describes
  your last commit on *this* machine. A manifest is a content-addressed
  snapshot (path → blob-SHA) of the whole tree at that commit.

One is shared:

- **Remote HEAD** — a single mutable pointer on the bucket
  (`<bucket>/<prefix>/HEAD`) holding the SHA of the current tip manifest.
  This is the only mutable file on the remote. Everything else
  (`manifests/<sha>`, `blobs/<sha>`) is write-once.

Every push rewrites remote HEAD. Every no-arg pull reads it.

Each clone of the repo (each machine) also has a `client_id` — a short
random hex string generated by `shasync init`, stored in
`.blobs/config.json`. It's the label that shows up in conflict-copy
filenames, so when you see a `(modified on … by 3f2a1c8e)` file you know
which machine produced it.

## 2. The normal day

```
shasync pull          # sync with the remote (merges if anyone else pushed)
<edit some files>
shasync push          # publish your changes
```

That's it. Two commands on the critical path. `pull` and `push` each do an
**implicit commit** of any uncommitted working-tree changes — so there is
no separate `commit` step you need to remember. (You can still run
`shasync commit -m "..."` if you want an intermediate checkpoint with a
meaningful message; it's useful for `log`, but nothing requires it.)

**What each command actually does:**

- **`shasync push`** — *First*, if the working tree differs from local
  HEAD, snapshot it into a new manifest with an auto-generated message
  (`sync from <client_id>`) and advance local HEAD. *Then* read remote
  HEAD. If it's empty, equal to local HEAD, or an ancestor of local HEAD,
  upload any new blobs + manifests and rewrite remote HEAD. If it's
  anything else (the remote moved while you were away), push refuses and
  tells you to run pull. Push never mutates the working tree — only the
  local commit chain.

- **`shasync pull`** — *First*, if the working tree differs from local
  HEAD, implicit-commit it the same way (so your in-progress edits become
  a real commit on the local chain). *Then* read remote HEAD and classify:
  plain fast-forward, already up to date, local is ahead (nothing to
  pull), or diverged (auto-merge — see §3). In the merge case, pull
  builds a merge manifest whose parent is the remote tip, materializes
  it into the working tree, and advances local HEAD.

- **`shasync commit -m "..."`** — Snapshot the working tree into a
  manifest with your message and advance local HEAD. Pure local; no
  network. Optional on the critical path; useful when you want an
  explicitly-titled commit for `log` to show later.

**A picture of the chain on one machine:**

```
(older) ←── sync from a1b2c3d4 ←── "refactor parser" (explicit) ←── sync from a1b2c3d4 ←── HEAD
```

Explicit commits and implicit commits are the same kind of object on
disk; only the `message` field differs. `shasync log` shows both.

## 3. The divergent case — offline edits on two machines

The only case where anything interesting happens is when two machines both
edit the same file without a sync in between. Example:

```
Monday:    A and B are both at commit c1.
Tuesday:   A edits hello.txt ("A version"), pushes.  (A is now at c2a)
           B is offline, edits hello.txt ("B version"),
           also edits sub/a.txt.
Wednesday: B comes online and runs `shasync push`.
```

When B runs push:

1. Implicit-commit B's working tree — a new local commit `c2b` whose
   parent is `c1`, carrying both of B's edits.
2. Read remote HEAD = `c2a`.
3. Classify: `c2a` is neither equal to `c2b` nor an ancestor of `c2b`.
4. **Refuse.** Print something like:
   ```
   remote has moved (remote=c2a..., local=c2b...) — run: shasync pull
   ```

B then runs `shasync pull`, which does the real work:

1. Implicit-commit any further working-tree changes (likely none this
   time — B already committed via push's failed attempt).
2. Fetch the remote chain back to the common ancestor (`c1`).
3. Build a **merge manifest** whose parent is the remote tip (`c2a`),
   combining the two sides file by file:
   - `sub/a.txt` — only B touched → kept at B's version.
   - `hello.txt` — both touched, differently → **conflict**:
     A's version (the remote) stays at `hello.txt`; B's version is
     stashed as a sibling copy:
     ```
     hello (modified on 2026-04-20 101530 by <B's client_id>).txt
     ```
   - Files neither side touched pass through unchanged.
4. Check the merge out into B's working tree and advance B's local HEAD.

Now B is at a merge commit whose parent is the remote tip. A second
`shasync push` is a plain fast-forward — uploads the merge manifest and
any new blobs, rewrites remote HEAD.

A, on the other machine, will get the merged state (including the
conflict-copy file) the next time it runs `shasync pull`.

**Key property:** nothing is ever lost. Both chains are real and reachable
on both machines — B's pre-merge commit `c2b` still exists on disk as an
orphan manifest under `.blobs/manifests/`, and B's version of `hello.txt`
is preserved as a real file in the merge. You resolve the conflict by
hand: edit `hello.txt` to the final version you want, delete the
`(modified on …)` sibling, push again.

## 4. What happens if I forget to pull

Nothing disastrous. Your edits are always safe:

- You can edit freely without pulling first. Push and pull both
  implicit-commit on the way in, so nothing is lost.
- If the remote moved while you were editing, push refuses — it never
  silently clobbers upstream work. You then run pull, which either
  fast-forwards (if you touched different files) or merges with
  conflict-copies (if you touched the same files).

The only habit worth developing is: if you're about to do a long session
of work on machine B after using machine A recently, run `pull` first to
start from the latest state. Not doing so is survivable — but you'll get
a conflict copy for every file both machines touched, which is more
cleanup than needed.

## 5. Guard-rails

- **Push never silently ignores uncommitted edits.** Before every push,
  the working tree is snapshotted. If you edit, save, push — the saved
  version is what ends up on the remote.
- **Pull never silently clobbers uncommitted edits.** Before every pull,
  the working tree is snapshotted too. If you edit, save, pull — your
  edits become a commit on the local chain *before* the merge sees them.
  In the worst case (both sides edited the same file) your edit ends up
  as a `(modified on … by <client_id>)` sibling copy, never deleted.
- **Push is read-only for the tree.** The only mutation push makes on
  your disk is the implicit commit. No working-tree rewrite happens on
  push — merges are pull's job.
- **Everything is content-addressed.** Redundant uploads are no-ops; if
  you push and retry, the second run does nothing new.
- **Blobs and manifests are immutable.** The only thing that can be
  "rewritten" is the remote HEAD pointer, which happens atomically on
  push. Concurrent pushes from two machines race last-writer-wins on
  that pointer — under the single-user-many-machines assumption that's
  acceptable (the losing machine will just get "remote has moved — run
  pull" next time and merge through).
- **No ghost commits.** An implicit-commit pass that finds zero file
  differences does nothing — local HEAD is unchanged. So `push`, `pull`,
  or `commit` on a clean tree won't pollute your log with empty entries.

## 6. Command cheat sheet

| Command                          | What it does                                                                              |
|----------------------------------|-------------------------------------------------------------------------------------------|
| `shasync init`                   | Create `.blobs/`, generate a `client_id`.                                                 |
| `shasync remote set <url>`       | Point this clone at an S3 or GCS bucket/prefix.                                           |
| `shasync pull`                   | Implicit-commit, then sync working tree (merges if diverged).                             |
| `shasync pull <sha>`             | Download-only. No implicit commit, no merge, no working-tree mutation.                    |
| `shasync push`                   | Implicit-commit, then fast-forward remote HEAD. Refuses otherwise.                        |
| `shasync push --force`           | Overwrite remote HEAD, skipping the topology check. Recovery only.                        |
| `shasync commit -m "..."`        | Explicit snapshot with a message. Optional checkpoint; not needed on the critical path.   |
| `shasync status`                 | Show which files differ from local HEAD.                                                  |
| `shasync log`                    | Walk your local manifest chain (implicit and explicit commits both appear).               |
| `shasync log --remote`           | Walk the remote's chain without updating your working tree.                               |
| `shasync info`                   | Summary: remote, HEAD, client_id, object counts.                                          |

## 7. Frequently-implicit worries

**"Will auto-commit pollute my log with garbage?"** Only if you push or
pull on a dirty tree, and the resulting commit is exactly the work you
just did. Clean-tree syncs are no-ops — no commit, no log entry. If the
`sync from <client_id>` messages in `log` bother you, run
`commit -m "..."` before `push` to give the snapshot a meaningful title
instead.

**"What counts as 'dirty'?"** Anything that makes `walkWorkingDir`
produce a manifest different from local HEAD: added files, removed
files, files whose size/mtime changed. Content-preserving touches of
mtime will trigger a re-hash but not a new commit (the hash will match).

**"Can I still work fully offline?"** Yes — just don't push or pull.
`commit -m "..."` is pure local, as are `checkout` and `status`. You
can accumulate commits locally for days; when you come back online,
`push` will either fast-forward (if no one else pushed) or refuse and
prompt you to pull (which will merge cleanly or with conflict-copies).
